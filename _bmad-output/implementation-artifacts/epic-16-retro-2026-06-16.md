## What Went Well

1. **Epic 15's three carry-forward action items all landed in Epic 16, on schedule.** AI#3 → Story 16.0 (`--check`/`--force`); AI#2 → per-story bootstrap regen with 16.4 idempotence-verify (Rule #24, no deferred bump); AI#4 → every destructive smoke asserted a guarded rejection (Rule #26). The retro→Story-X.0 feedback loop worked exactly as designed.
2. **The Story 16.0 footgun fix proved itself within the same epic.** During the 16.0 code review, a sub-agent ran the *old* generator with `--check` (unknown flag → fell through to the write path) and clobbered the frozen `governance-baseline.ts`; `git checkout` restored it. That incident is live proof the `--check`/`--force` guard (Rule #25) addressed a real, reachable hazard — and after the fix landed, the same mistake is impossible (the write path refuses without `--force`).
3. **Empirical IRIS probing corrected the plan twice with zero fabrication** (Rules #2/#16). Both wrong epics.md class/API claims were caught BEFORE shipping a half-working tool, amended in the planning artifact in place (Rule #5), and flagged for this retro — `restore` deferred honestly rather than faked.
4. **Adversarial 3-layer review caught handler bugs the unit tests structurally couldn't.** 16.1's comma-injected action (`"suspend,resume"`) bypassing a substring-match dispatch and the fail-OPEN guard on `%OpenId` failure; 16.3's listHistory aborting on a single corrupt `^SYS("BUHISTORY")` node. All were on direct-REST paths the mocked-fetch unit tests don't reach — caught, fixed, redeployed, live-verified.
5. **Governance frozen-foundation model held cleanly across +3 tools / +14 keys.** New ops keys stayed out of the frozen baseline, governed by `mutates` + `defaultSeed`; `--check` reported them as allowed post-foundation; hash unchanged. Rule #23 held without friction.
6. **Live destructive-path rejections confirmed the guards on the running server** (Rule #26): terminate of the CONTROL daemon refused; dismount/truncate of a non-existent directory refused; `restore` not-supported; `run` without `taskName` rejected — all changing nothing.

## What Could Be Better (watch-items)

1. **The `.vscode/settings.json` IDE-sync toggle is a recurring sub-agent trap.** Dev/CR sub-agents that deploy to live IRIS toggle `objectscript.conn.active` to `false` (per the user's global bulk-export rule) and don't always restore it; once a sub-agent **staged** the toggled value into the git index. The lead caught and restored/unstaged it before every commit (it never reached a commit), but it cost repeated detection cycles. **Not a project-rule-worthy pattern** (per the Lead's call — it's covered by the existing user-CLAUDE.md toggle guidance and the epic-cycle Window mechanics; the gap is sub-agent discipline) → captured as Action Item #2 (spawn-prompt hardening), not a rule.
2. **epics.md carried two wrong IRIS API claims into Epic 16** (the 16.2/16.3 corrections above). The mid-epic Rule #16 probe caught both, but pre-spec probing would have avoided the in-flight amendments. Already a known pattern (Rule #16); no new rule — affirm the existing discipline and probe at story-creation time for Epic 17's IRIS-class-dependent stories.
3. **Per-story bootstrap version churn (expected).** `BOOTSTRAP_VERSION` moved 4× (16.1/16.2/16.3-dev/16.3-CR) before 16.4's idempotence check — correct per Rule #24 (it's a content hash), documented so it isn't mistaken for drift.
