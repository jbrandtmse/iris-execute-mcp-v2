## What Went Well

- **Smoke-first hardening was the epic's headline win (Project Lead's call).** The include-now fixes were four ObjectScript guards on the new opt-in `iris_production_item` add/arbitrary-settings surface (CR 17.2-1/-2/-3/-5). These are server-side guards that the mocked-HTTP TypeScript unit tests **structurally cannot exercise** (Rule #26). The lead's live-HTTP per-story smoke drove all five guarded paths against the deployed endpoint on HSCUSTOM — bogus className, duplicate item name, unknown `@`-suffix, leading-`@` empty name, and the `||` IdKey delimiter (CR 17.1-1 re-verification) — and every one was rejected with `result:{}`. A follow-up `SELECT … WHERE Name %STARTSWITH 'ZZZSmoke'` returned **0 rows**, mechanically proving the rejections changed nothing. The smoke didn't just confirm the happy path; it *was* the coverage for the security-critical guards.
- **Triage discipline held.** Every carried Epic 16–17 deferred item was explicitly routed (include-now / re-defer-with-rationale / drop), and AI#4 (backup-restore) was re-affirmed deferred. The ledger in `deferred-work.md` is current — accumulation did not go silent (the Rule #1 / retro→Story X.0 loop, now functioning across Epics 14–18).
- **Strictly-additive guarantee preserved (Rule #19).** The new guards reject *previously-silently-accepted bad input* on the default-disabled opt-in surface; the existing `enable`/`disable`/`get`/`set` + 6 working keys stayed byte-for-byte (the Story 17.2 back-compat `toEqual` snapshots remained green and unweakened). Bootstrap re-rolled per Rule #24; frozen governance baseline untouched (Rule #23/#25).
- **Code review correctly dismissed a false HIGH.** The Edge Case Hunter flagged "set @-key like `user@example.com` now rejected" as a back-compat regression; it was dismissed after verifying the arbitrary-`@` routing was *introduced* in Story 17.2 (not pre-governance grandfathered) and that `Ens.Config.Setting` names cannot contain `@` — so the old acceptance silently created an unconsumable setting. Rejecting it is strictly-better.

## What Was Challenging

- **Documentation governance-callout gap (process finding → Rule #30).** The per-epic docs rollups (Stories 15.6, 16.4, 17.4) reliably added tool **catalog rows** and refreshed **counts**, but only Epic 15's rollup added an explicit **governance default-disabled callout**. Epics 16 and 17's new write tools/actions were listed without indicating which were default-disabled under `IRIS_GOVERNANCE`. The Project Lead caught this at Epic 18 close-out and requested a doc pass: README (new "Default-disabled write actions" table), `tool_support.md` (Epic 16/17 governance notes), and every per-server README now flag the default-state. This is a recurring-shape gap — it would silently recur on every future governed-tool epic — so it is codified as **Rule #30**.
