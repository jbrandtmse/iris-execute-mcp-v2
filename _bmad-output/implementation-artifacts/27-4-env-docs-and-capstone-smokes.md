## Lead Capstone Finding (2026-07-11) — HIGH: mappings-diff key includes the per-side namespace

**The AC 27.4.2 live capstone (authorized real `execute` WRITE) FAILED its re-diff-clean step — surfacing a genuine HIGH correctness bug in the `mappings` domain (Story 27.1 code, committed `3a07d8f`) that mocked same-namespace fixtures + the drift-COUNTING earlier smoke could not catch.**

**What happened (live, HSCUSTOM `default` → SADEMO `sademo`):** seeded a disposable `ZENVCAPSTONE → HSLIB` package mapping on HSCUSTOM only → `iris_env_diff` detected it `onlyInSource` (`package::HSCUSTOM::ZENVCAPSTONE`) → `plan` → `execute` (step allowlisted, `confirm:true`) reported `completed` and **genuinely created the mapping on SADEMO** (authoritatively confirmed via `iris_mapping_list`: SADEMO 8→9 package mappings, `ZENVCAPSTONE→HSLIB` present) → BUT the re-diff STILL showed `ZENVCAPSTONE` as drift: `onlyInSource:[package::HSCUSTOM::ZENVCAPSTONE]` AND `onlyInTarget:[package::SADEMO::ZENVCAPSTONE]`. (Seeded item cleaned up from BOTH namespaces afterward.)

**Root cause (`packages/iris-dev-mcp/src/tools/env-diff.ts:523-524`):** `mappingKey(entry) = \`${entry.type}::${entry.namespace}::${entry.name}\`` includes the per-side **config namespace**. Since the source enumerates its namespace (HSCUSTOM) and the target enumerates its namespace (SADEMO), the SAME logical mapping gets DIFFERENT keys per side (`package::HSCUSTOM::Ens` vs `package::SADEMO::Ens`) → the two sides can NEVER match → every mapping that is genuinely identical across the two namespaces is reported as spurious `onlyInSource` + `onlyInTarget` drift, and `promote` can NEVER make a mapping "clean" (the promoted item keeps a different key on each side). **The "145 mappings drift" celebrated in Story 27.1 was largely this namespace-key ARTIFACT, not genuine drift** (only `HS.Local` + `SessionAgent` are truly HSCUSTOM-only; the other ~8 package mappings exist identically in both but were mis-reported as drift). The namespace is the SIDE being compared, NOT part of the item's identity — it must NOT be in the comparison key. This manifests whenever `source` and `target` resolve to DIFFERENT namespace names (the accepted two-profiles-one-instance test shape AND any real stage/prod deployment using different namespace names).

**Scope check — ONLY mappings is affected:** SDS key = `production||item||hostClass||setting` (no namespace), webapps key = `name`, config key = property name, documents key = doc name — none embed the per-side namespace. `mappingValue` (`:527-531`) correctly excludes namespace (compares only `database`/`collation`/`lockDatabase`), so the fix is isolated to the KEY.
