## What Went Well

1. **Story 2.0 resolved all Epic 1 MUST-INCLUDE items before feature work** — outputSchema Zod type fix, CSRF preflight on first POST, headRequest refactoring via executeFetch(). Zero issues propagated to downstream stories.

2. **Clean tool delivery cadence** — Each story built cleanly on shared infrastructure. Stories 2.2-2.7 delivered tools incrementally without breaking changes or circular dependencies. No story required rework of another story's code.

3. **Code review caught 17 substantive issues** — All patches were 1-3 line fixes (batch delete partial failure, URL encoding, input validation, switch exhaustiveness, type safety). None cascaded. Quality maintained without slowing delivery.

4. **Integration test infrastructure delivered** — Story 2.8 created reusable patterns: IRIS availability detection, API version negotiation in setup, version-gated test skipping. 19 integration tests cover all 13 tools.

5. **Upstream bug discovered and fixed** — Integration tests in Story 2.8 uncovered that `negotiateVersion()` was reading `result.version` (IRIS build string) instead of `result.content.api` (numeric API version). Fixed in shared package, benefits all downstream epics.

6. **Monotonic test coverage growth** — Tests increased every story: 118 → 126 → 145 → 152 → 163 → 184 → 199 → 220 → 258. No test debt accumulated.

## What Didn't Go Well / Challenges

1. **Atelier API endpoint research per story** — Each story required cross-referencing vscode-objectscript source for endpoint URLs, methods, body formats, and version requirements. A consolidated Atelier API reference document would have eliminated this repeated research.

2. **Test helper duplication across test files** — `createMockHttp`, `createMockCtx`, and `envelope` helpers were redefined in doc.test.ts, compile.test.ts, sql.test.ts, and server.test.ts. Deferred but creates maintenance burden.

3. **Integration tests delivered at end of epic** — Waiting until Story 2.8 delayed finding the negotiateVersion bug. Story-by-story integration tests would have caught it earlier.

4. **Tool edge case patterns matured gradually** — Stories 2.2-2.4 established core patterns, but reviews kept finding similar edge cases (empty arrays, silent failures, inconsistent query construction). Patterns stabilized by Story 2.5+.

## Lessons Learned

1. **Story-0 pattern for deferred cleanup works** — Dedicating a story to resolving previous epic's deferred items before feature work prevented issue propagation. Adopt for all future epics.

2. **Integration tests should ship per-story, not end-of-epic** — Story 2.8's late integration tests missed the negotiateVersion bug until the end. Earlier integration testing catches upstream issues sooner.

3. **Tool implementation patterns need upfront documentation** — Edge cases (empty arrays, partial failures, URLSearchParams consistency) were caught reactively in reviews. A "Tool Implementation Checklist" would prevent these proactively.

4. **Atelier API research should be consolidated upfront** — Each story's handler required researching vscode-objectscript for endpoint details. A one-time extraction into a reference document would eliminate this per-story overhead.

5. **Code review as continuous quality gate is effective** — 17 patches across 9 stories maintained quality without rework. All were small, localized fixes. The three-layer review pattern (blind hunter, edge case hunter, acceptance auditor) catches real issues.
