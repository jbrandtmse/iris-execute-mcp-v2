## What Went Well

- **E1 "single source of truth" paid off immediately.** The roster is built from `profiles.ts` and the policy from the shared `getEffectivePolicy` — the IDENTICAL call the D6 governance resource uses. The lead smoke and a QA e2e test both assert the discovery policy `toEqual`s the live D6 `resources/read` for the same profile, so the tool and the resource **mechanically cannot drift**. DRY + drift-free, exactly as the ADR intended.
- **The live MCP-reload test was the headline validation (Project Lead's call).** After the user reloaded the servers, calling `iris_server_profiles` on all five surfaced what mocked unit/e2e tests structurally cannot: each server reports its **own** governed surface with correct default-disabled writes — admin (`iris_service_manage:enable/disable/set`, `iris_audit_manage:purge`…), ops (`iris_database_action:*`, `iris_backup_manage:run/freeze/thaw`), interop (`iris_production_item:add/remove`, `iris_default_settings_manage:set/delete`) all `false`, reads `true`. This is the cross-server reality the epic set out to make discoverable, proven against the real wire.
- **Source-grounded story spec (Rule #16) → clean dev landing.** The story was written against the actual `server-base.ts`/`profiles.ts`/`governance.ts`, so the dev's special-case-in-`handleToolCall` design (discovery reports in-memory config and never connects — works even when IRIS is down) landed without rework. The connection-agnostic design is a genuine win: an agent can discover its environment before any IRIS is reachable.
- **Strictly-additive guarantee held (Rule #19/#23).** No `BOOTSTRAP_VERSION` change (no ObjectScript), frozen baseline untouched, default-only (no `IRIS_PROFILES`/`IRIS_GOVERNANCE`) back-compat proven mechanically — the new tool is purely additive on top of today's behavior.
- **Zero-debt close.** At the Project Lead's request, both CR 19.0 LOW findings were resolved during the retro (not deferred), so Epic 19 closes carrying no deferred items.

## What Was Challenging

- **Framework-tool counting shape (process finding → Rule #31).** Registering the tool centrally (not in any package `tools/index.ts`) broke 13 pre-existing count/shape assertions: advertised `toolCount`/`getToolNames` and policy-map-size cross-checks DID move (+1 per server), while package-array `index.test.ts` lengths correctly stayed unchanged. The dev resolved it correctly (update advertised/cross-server expectations; document the tool as a framework surface; leave package-array lengths alone), but it is a recurring shape — every future framework capability hits the same two-count split. Codified as **Rule #31**.
- **Story-19.0 naming collision with the retro-review gate.** The `/epic-cycle` retro-review gate normally creates "Story N.0" for the previous epic's deferred cleanup, but Epic 19's substantive feature story was already 19.0. Resolved cleanly: Epic 18's carried items are ObjectScript/ops hardening routed (by the Epic 18 retro) to a future *cleanup* epic, so they were re-deferred and no colliding Story X.0 was created. It is a recurring tension when a feature epic immediately follows a cleanup epic — handled by triage, no rule warranted.
