## What Went Well

1. **Fully automated pipeline** — The entire epic ran hands-off: story creation, development, code review, commit, and push all automated via the epic-cycle workflow with zero user intervention required.
2. **Architecture document quality** — Exact interfaces (ToolDefinition), file paths, naming conventions, and anti-patterns specified upfront eliminated ambiguity. No clarification requests were needed.
3. **Incremental test coverage** — Tests built up story-by-story: 0 → 37 → 66 → 112 → 125. No test debt accumulated.
4. **Code review effectiveness** — Reviews caught 6 real issues (missing .prettierignore, dead-code agents, console.warn in ESLint allow list, non-JSON response handling, port validation, JSDoc inaccuracies) before they could compound.
5. **Clean story sequencing** — Each story built cleanly on the previous. Story 1.2 (HTTP client) enabled 1.3 (health/atelier), which enabled 1.4 (server base), which enabled 1.5 (integration tests).

## What Didn't Go Well / Challenges

1. **13 deferred work items accumulated** — While individually LOW severity, several create risk for Epic 2 if not addressed (see Action Items below).
2. **headRequest code duplication** — The `head()` method in IrisHttpClient duplicates ~90% of the `request()` method. Future auth or error handling changes must be applied in two places.
3. **Empty package test noise** — `turbo test` shows failures for packages with no test files (iris-admin-mcp, iris-ops-mcp, etc.). Cosmetic but creates noise.
4. **CSRF token gap on first mutating request** — If a consumer's first HTTP call is POST/PUT/DELETE, no CSRF token has been extracted yet. IRIS may reject the request.

## Action Items

### MUST-INCLUDE in Story 2.0 (Epic 1 Deferred Cleanup)

These 3 items MUST be resolved before Epic 2 story work begins. They directly impact Epic 2's Atelier API tools.

1. **Fix outputSchema type mismatch** — `ToolDefinition.outputSchema` is typed as `object` (plain JSON Schema) but MCP SDK's `registerTool` expects Zod. Epic 2 tools that provide outputSchema will hit this. Resolution: either change the interface to accept Zod, or add a conversion layer in McpServerBase.
   - Source: deferred-work.md, code review of 1-4
   - Impact: Blocks any tool that declares outputSchema

2. **Refactor headRequest to eliminate code duplication** — Extract shared fetch/auth/cookie/error logic from `request()` and `headRequest()` into a common private method. This prevents divergence when auth behavior changes in Epic 2.
   - Source: deferred-work.md, code review of 1-3
   - Impact: Maintenance risk — any auth or error handling change must be applied in two places

3. **Fix CSRF token missing on first POST** — Add an explicit `connect()` method or lazy-init GET to ensure a CSRF token is available before the first mutating request. Epic 2 has PUT/POST tools (iris.doc.put, iris.compile.do) that could fail without this.
   - Source: deferred-work.md, code review of 1-2
   - Impact: Atelier API PUT/POST tools may fail on first call

## Lessons Learned

1. **Comprehensive architecture specs enable full automation** — When interfaces, file paths, and patterns are specified precisely, the dev-review-commit cycle can run without human intervention.
2. **Code review deferred items need severity triage** — 13 items accumulated but most are LOW. The 3 that matter for the next epic should be separated from cosmetic concerns.
3. **Integration test infrastructure pays off early** — Story 1.5's test helpers and skip patterns will be reused by every subsequent epic's integration tests.
4. **Story sequencing matters** — Building shared infrastructure first (config → HTTP → health → server base → tests) created a stable foundation with no circular dependencies.
