## What Went Well

1. **Story 3.0 pattern continues to work** — Resolving all Epic 2 deferred items before feature work prevented issue propagation. Third consecutive epic using this pattern successfully.

2. **SqlProc breakthrough for bootstrap** — Adding `[SqlProc]` to Setup.Configure() and IsConfigured() solved the chicken-and-egg problem elegantly. The bootstrap can call these via the existing Atelier SQL endpoint without needing the custom REST service to already be running.

3. **Dev agent proactive implementation** — The Story 3.3 agent built most of Story 3.4 (UnitTest handler, TypeScript tool, tests) without being asked, saving an entire story cycle.

4. **Consistent handler pattern** — The namespace-switch/try/work/render/catch/sanitize/restore pattern from Story 3.1 was reused cleanly across all 3 handler classes (Command, Global, UnitTest) with zero architectural rework.

5. **Mid-epic course correction** — Stories 3.8 (timeout config) and 3.9 (bulk load) were added smoothly via correct-course. Neither disrupted existing work.

6. **Live testing caught 9 critical bugs** — The decision to pause the retro and test the actual MCP server was the single most valuable action in this epic. These bugs were invisible to unit tests.

7. **Atelier async work queue for test execution** — Switching from custom REST to the Atelier `/work` endpoint (following the vscode-objectscript pattern) was the right call. Battle-tested, no I/O redirect issues, structured results.

8. **Successful v1 server replacement** — All 21 tools verified working. Both old MCP servers (iris-execute-mcp v1 and mcp-server-iris) fully decommissioned. Zero dependency on legacy code.

## What Didn't Go Well / Challenges

1. **6 stories shipped without code review** — Code review cadence broke after agent spawn issues in Story 3.4 and was never recovered until caught in the retro. The reviews eventually found a SQL injection bug (Story 3.6) and stale test assertions (Story 3.4).

2. **Agent spawn hangs** — Story 3.4 dev agent failed to spawn 3 times. Root cause never identified (possibly nested Skill tool invocation depth or MCP connection saturation). Work completed inline instead.

3. **9 bugs found only through live testing** — None were caught by unit tests or code review:
   - `AutheEnabled=64` is Kerberos, not Password (32) — wrong auth method on web app
   - `%request.Content` is MultiDimensional in `%CSP.Request` — use `GetMimeData("BODY")`
   - Double JSON response from `RenderResponseBody` + `%Atelier.REST` error handler — set `tSC=$$$OK` after rendering
   - Web app namespace must match where classes are compiled (HSCUSTOM, not USER)
   - `%UnitTest.Manager.RunTest` with `/noload` needs colon-prefix testspec and `/norecursive`
   - `^UnitTestRoot=""` falls back to IRIS source dir — need real temp directory
   - Atelier `/work` endpoint returns job ID in `Location` HTTP header, not response body
   - Atelier poll returns empty `result: []` while tests still running — must wait for non-empty array
   - CSP gateway doesn't recognize programmatically created web apps — must save via SMP

4. **Web app gateway registration** — `Security.Applications.Create()` does not notify the CSP gateway. The web app must be saved through the Management Portal or the gateway needs a restart. This is a fundamental IRIS operational constraint that affects the bootstrap flow.

5. **TypeScript build errors after Story 3.8** — Adding `timeout` to `IrisConnectionConfig` broke test files that constructed config objects without it. Caught only at build time, not by test runner (Vitest doesn't run tsc).

6. **MCP client type serialization** — Boolean parameters sent as strings (`"true"` instead of `true`) caused validation errors. Fixed with `booleanParam` Zod preprocessor and improved parameter descriptions.

## Lessons Learned

1. **Live testing is non-negotiable** — Unit tests with mocked HTTP give false confidence. The 9 bugs found through live testing would have blocked every user on first connection. Add a live smoke test step to the epic cycle pipeline.

2. **Code review must be a mandatory gate** — When the pipeline is disrupted (agent hangs, mode switches), code review is the first thing dropped. It should require explicit user approval to skip, not silently omit.

3. **IRIS web app creation requires gateway awareness** — Programmatic creation via `Security.Applications.Create()` is necessary but not sufficient. The CSP gateway must be notified. Document this as a known limitation and include SMP save in the setup instructions.

4. **Atelier API is the right default** — Using Atelier endpoints (SQL for bootstrap, `/work` for tests) is more reliable than custom REST handlers for system operations. Reserve custom REST for operations Atelier genuinely can't do (command execution, global operations, I/O capture).

5. **`%CSP.Request.Content` is MultiDimensional** — This is an ObjectScript platform constraint that isn't documented prominently. Use `%request.GetMimeData("BODY")` to safely access POST body streams in REST dispatch handlers.

6. **Boolean/numeric Zod coercion is essential for MCP** — MCP clients may serialize non-string types as strings. Always use preprocessors (`booleanParam`, `z.coerce.number()`) for non-string parameters.

7. **Embedded class content drifts** — The `bootstrap-classes.ts` file embeds ObjectScript source that can become stale as the `.cls` files are updated. Need a regeneration step or build-time sync.
