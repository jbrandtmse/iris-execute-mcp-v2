# Story 14.3: Governance Policy Model, Action Classification & Cascade Resolution

Status: done

## Story

As an operator protecting an instance from unwanted mutations,
I want a policy model that enables/disables tool actions with a global baseline and per-profile overrides,
so that I can lock down writes globally and tune exceptions per environment, with safe defaults.

## Acceptance Criteria

1. **AC 14.3.1** ‚Äî Each tool action carries `mutates: 'read' | 'write'` classification metadata declared at registration. Per D4, only **NEW** actions (Epics 14‚Äì17) carry it; existing tools are grandfathered (no retro-classification). The metadata lives on `ToolDefinition` (optional field) so future governed tools declare it co-located with the tool.
2. **AC 14.3.2** ‚Äî `@iris-mcp/shared` parses `IRIS_GOVERNANCE` (JSON: `{ "global": { "<tool|tool:action>": true|false }, "profiles": { "<name>": { ... } } }`) at startup; malformed JSON fails fast with a clear error **naming `IRIS_GOVERNANCE`** (mirror the `profiles.ts` / `loadConfig` fail-fast style).
3. **AC 14.3.3** ‚Äî The **default seed**: existing actions = enabled; new `read` actions = enabled; new `write` actions = **disabled**. "Is this action new?" is derived from membership in the generated baseline (AC 14.3.7) ‚Äî in-baseline ‚áí grandfathered enabled; not-in-baseline ‚áí new (read enabled / write disabled via its `mutates` metadata).
4. **AC 14.3.4** ‚Äî Effective resolution implements `effective(key, profile) = profile.explicit(key) ?? global.explicit(key) ?? defaultSeed(key)`. A profile override may enable or disable, in either direction.
5. **AC 14.3.5** ‚Äî Resolution is unit-tested across: default-seed only, global-enable of a new write action, profile override down (disable globally-enabled), profile override up (re-enable globally-disabled), silent-profile-inherits-global, and unknown-action handling. (Use synthetic new-action fixtures ‚Äî Epic 14 adds no real new actions.)
6. **AC 14.3.6** ‚Äî A `getEffectivePolicy(profile)` API returns the full enabled/disabled action map for a profile (consumed by 14.4 enforcement and 14.5 resource).
7. **AC 14.3.7** ‚Äî **Provable back-compat baseline (D3):** a build-time generator `scripts/gen-governance-baseline.mjs` (mirroring `gen-bootstrap.mjs`) enumerates every existing tool and its `action`-enum values into a checked-in `packages/shared/src/governance-baseline.ts` (**generated, output-only ‚Äî Rule #18; header `// DO NOT EDIT ‚Äî auto-generated by scripts/gen-governance-baseline.mjs. Regenerate via: pnpm run gen:governance-baseline`**). The default seed derives "is this action new?" from membership in this baseline. A `gen:governance-baseline` script is added to `package.json` and run wherever `gen:bootstrap` drift is checked. A test asserts that, under an **empty** `IRIS_GOVERNANCE`, **every** baseline action resolves to enabled ‚Äî the mechanical proof of "no existing capability disabled by default."

## Integration ACs

**Service-introducing story ‚Äî no consumer in this story.** Story 14.3 introduces the governance policy engine (`governance.ts`: parse, cascade, `getEffectivePolicy`) + the generated baseline. **The first consumer will be Story 14.4** (call-time enforcement in `handleToolCall`), with Story 14.5 (the governance resource) a second consumer. This story's verification is the AC 14.3.5/14.3.7 unit suite. Satisfies the epic-cycle Integration-AC gate escape clause (producer with no consumer yet in this story).

## Dev Notes

### Architecture decisions in scope (design authority ‚Äî [architecture.md](../planning-artifacts/architecture.md) "Multi-Server Profiles & Tool Governance")

- **D3 ‚Äî Provable back-compat via a generated governance baseline.** Generator mirrors `gen-bootstrap.mjs`; enumerates every existing tool + `action`-enum values into `governance-baseline.ts` (generated, output-only). Seed enables every baseline entry; non-baseline ‚áí new (read enabled, write disabled). Makes "no pre-existing action disabled by default" mechanically verifiable, not a hand-maintained `isNew` flag.
- **D4 ‚Äî Governance key + `action` discriminator.** Key is `tool` for single-op tools, `tool:action` for multi-action tools. New governed tools surface mutations via an `action` enum; only NEW actions carry `mutates`. Existing single-op tools grandfathered.
- **D7 ‚Äî Config parsing & fail-fast.** Parse `IRIS_GOVERNANCE` centrally at startup; malformed JSON fails fast naming the var. (Mirror the 14.1 `profiles.ts` fail-fast pattern ‚Äî `loadProfileRegistry` is the precedent.)

### Epic-14 reality check (important)

Epic 14 introduces **no new IRIS tools** ‚Äî it is the foundation. So the generated baseline contains **all** current tools/actions, and under an empty `IRIS_GOVERNANCE` the seed enables **everything** (the back-compat gate). The `mutates`/new-action machinery is exercised by **synthetic test fixtures** in this story; real new write actions arrive in Epics 15‚Äì17 (where the baseline correctly classifies them as "new"). Do not invent `mutates` classifications for existing tools.

### Enumeration specifics (verified)

- Each server package: `export const tools: ToolDefinition[]` in `packages/<pkg>/src/tools/index.ts` ‚Üí built to `dist/tools/index.js`. The 5 packages: `iris-dev-mcp`, `iris-admin-mcp`, `iris-interop-mcp`, `iris-ops-mcp`, `iris-data-mcp`.
- Multi-action tools: `action: z.enum([...])` inside `inputSchema` (e.g. `iris_user_manage` ‚Äî [user.ts:32](../../packages/iris-admin-mcp/src/tools/user.ts#L32)). Zod v4: a `ZodEnum`'s values are on `.options`. Single-op tools (e.g. `iris_user_get`) have no `action` field ‚Üí key is the bare tool name.
- `gen-bootstrap.mjs` patterns to mirror: CRLF‚ÜíLF normalization for hash stability, short SHA-256 content hash, single `writeFileSync`, console summary. The governance baseline is a Set of strings (sorted for deterministic output/diff).
