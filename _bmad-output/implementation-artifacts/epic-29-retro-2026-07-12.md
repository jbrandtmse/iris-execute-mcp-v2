## 2. What Went Well

1. **The scope-seam between 29.0 and 29.1 was clean (→ new Rule #52).** 29.0 shipped a *basic-but-complete-shape* outcome derivation with the deferred-fidelity seam explicitly named in its Dev Notes; 29.1 closed exactly that seam (structured `denyReason`/`presetApplied` copied verbatim, schema-aware `action`, per-session monotonic `seq` under concurrency, shutdown flush) without re-touching 29.0's finished writer/rotation code. Zero rework, no over- or under-building.
2. **QA earned its keep every story — no padding.** 29.0: outcome-derivation e2e through the real SDK callback. 29.1: a mixed-outcome concurrent batch proving conditional fields (`error`/`denyReason`/`presetApplied`) never bleed across queued entries. 29.2: codified the live-smoke redaction guarantee *through the real interceptor* with a Zod-declared `password` field (prior redaction tests were all unit-level).
3. **The adversarial review caught real, non-obvious defects.** 29.0: a MED where the *synchronous* audit work (redaction walk + `JSON.stringify`) could throw into the tool path (only the async write was guarded). 29.2: the README JSONL example documented a wrong `serverPkg` (scoped `@iris-mcp/ops`, from `options.name`) and included the stripped `server` key — a real trap for a compliance-log parser.
4. **The redaction guarantee held end-to-end.** Fail-closed substring key-family matching + 2 KB truncation, proven zero-secret-leakage across the three-nesting-position test, arrays/key-case fuzz, AND the built-dist lead smoke (grep the emitted file for the secret → 0 hits). The dev also surfaced (honestly, not glossed) that the MCP SDK strips undeclared keys before the interceptor sees `rawArgs` — a strictly *safer* property that required a genuinely-declared credential field to exercise the redaction path at all.
5. **The Rule #37 burn-down was disciplined and safe.** 41/41 items terminal with zero re-deferral; every resolved fix mutation-verified (8/8 sampled RED-on-revert across QA + review); the OS-vs-bootstrap-bump disposition policy (→ new context for future burn-downs) kept it TS+docs-only and additive — frozen baseline + `BOOTSTRAP_VERSION` untouched. Two carried items (CR 27.0-2/-5) were found already fixed in Story 27.1 (closed-with-evidence); two (CR 28.0-1/-2) closed with fresh live EXPLAIN probes.

## 3. What Could've Gone Better

1. **The burn-down's own disposition tally was arithmetically wrong (→ new Rule #51).** Story 29.3's dev claimed "20 resolved · 4 closed-with-evidence · 17 closed-by-decision"; a mechanical recount of the disposition column gave the true **14 · 4 · 23**. Every item was genuinely terminal (the substantive AC held), but the *summary* was hand-counted and off by 6. QA's grep-the-column recount caught it; the reviewer corrected the tally line + the story's Completion Notes. Lesson codified: never hand-author a count over a table.
2. **A QA mutation-verify near-miss.** The QA agent's first revert-restore used `git checkout -- <file>`, which discards the entire *uncommitted* Story 29.3 diff (not just the one-line mutation). Caught immediately via a diff-stat comparison and recovered from a pre-edit backup; all subsequent mutation-verifies restored from `/tmp` backups. No lasting damage — flagged so future mutation-verify passes on an uncommitted diff use file-copy backups, never `git checkout`.
