## What went well

1. **The /epic-cycle pipeline scaled cleanly across four stories.** Spawn-on-demand dev/review agents, in-line live verification, atomic per-story commits + chore-log commits — same shape as Epics 1–9, no drift.

2. **Story 10.1 set a pattern that 10.2 inherited cleanly.** The `truncated?: true` literal-type pattern for the package-list cap was reused verbatim in 10.4 for `skippedItemsTruncated?: true`. Helper-export-for-testability pattern from 10.1 (`stripDocExtension`, `rollupPackage`) was the template for 10.2's `docNameToFilePath`.

3. **Live verification caught real bugs unit tests missed.**
   - Story 10.1 live verification surfaced the CSP-path edge case (forward-slash doc names splitting into a stray `"2"` package row); patched in code review with the `(csp)` synthetic bucket and a regression test.
   - Story 10.2 stress test (HSCUSTOM, 13,277 docs in 82s) gave concrete proof the bounded-concurrency design held under load.
   - Story 10.4 was *born* from a live stress test of `%SYS` — 2,174 skipped items would have stayed invisible in the unit-test mocks, but the live response-envelope-overrun was unmistakable.

4. **The 2026-04-19 `IrisApiError.message` detail-formatter fix paid for itself twice.** Both Story 10.4 (the inline-`skippedItems` token-cap diagnosis) and the post-Epic `iris_execute_tests` `<SUBSCRIPT>UnitTestResultToJSON` investigation depended on real IRIS error text being surfaced. Without that prior fix, both diagnoses would have stalled at "IRIS reported errors. Review the error details."

5. **Story 10.4 was added cleanly via /bmad-correct-course mid-cycle without breaking the epic narrative.** Sprint Change Proposal → story file → dev/review/commit → log entry, all the same shape as the planned three stories.

6. **Code reviews found zero HIGH issues across the epic.** All MEDIUM and LOW items were auto-resolved in the same review session. Two notable patches:
   - 10.2 review caught a real data-loss surface — short-path collisions on Windows (two distinct doc names truncating to the same on-disk path) — and added a `reservedPaths: Map` guard.
   - 10.4 review confirmed both response paths (happy + hardError) were capped, the field-absent-not-false pattern was correct, and the manifest stayed uncapped.

7. **Auto-upgrade flow worked end-to-end multiple times.** Two `BOOTSTRAP_VERSION` changes during the epic; both rolled out cleanly via the version-stamp probe + redeploy/recompile path. No operator intervention required for any installed instance.

8. **Cross-namespace coverage was real.** Live verification of every story exercised at least USER + OPTIRAG (later: HSCUSTOM, %SYS). The cross-namespace path through the TypeScript HTTP client was the original concern when the project added the per-call `namespace` parameter; Epic 10 confirmed it remains solid.

## What could have gone better

1. **OPTIRAG namespace deletion mid-session was a real-world environment instability that interfered with retest agents.** The interop retest had 6 partial results because OPTIRAG was unreachable. Not a tool bug, but worth noting that test-agent prompts assumed OPTIRAG presence — when it disappeared, the partial results obscured the underlying tool health. Future test-agent prompts should fail-soft and substitute another namespace rather than reporting partials.

2. **`iris_doc_load` prefix-leakage bit us repeatedly during the epic.** I had to use the `Intero*.cls` glob workaround multiple times during Story 10.4 deploys because the running `iris-dev-mcp` server was always one MCP-restart behind the freshly-built code. The fix landed in the 2026-04-19 bug-fix pass, but in-session the running server kept the buggy version until each restart. **Lesson: when iterating on a TypeScript fix, mentally tag everything that calls it as "needs MCP restart before re-test."**

3. **`^UnitTestRoot` bug surfaced AFTER Epic 10 closed and required re-opening to fix.** The retest pass discovered `iris_execute_tests` was broken (despite working during Story 10.1 verification); root cause wasn't in our code but required a defensive bootstrap addition + per-call ensure. This was patched in commit `e7ad640` after the formal epic-cycle had closed. **Lesson: full-suite retest immediately after each epic's last story (before declaring done) catches this class of cross-epic regression earlier.**

4. **One subtle test mismatch from the per-call ensure.** The new ensureUnitTestRoot SQL POST in `iris_execute_tests` broke an existing assertion (`toHaveBeenCalledTimes(2)` → had to become `3`) and three bootstrap tests needed mock updates. All caught by `pnpm test`, but a reminder that adding "defensive" pre-call SQL to a tool is a behavior change observable from tests.

5. **`iris_doc_search` case-flag fix is hard to regression-test on a moving corpus.** Both retest passes had trouble verifying it because the corpus drifted (different match counts each session). A dedicated synthetic test fixture would be more stable than relying on `classmethod`/`ClassMethod` matches in the live IRIS dictionary.

6. **CSP-asymmetry observation deserved its README entry sooner.** I caught this during the %SYS stress test (2,174 CSP 404s out of 6,131 docs) and added the README note before Story 10.4 implementation, but ideally the README would have flagged this from the start of `iris_doc_export` — anyone exporting from `%SYS`-style namespaces hits it immediately.

## Lessons learned (carry to whichever epic comes next)

1. **Write live-verification fixtures into the story file itself.** Every Story 10.x file had a "live verification" section in the Tasks. That made it easy for test agents to know exactly what to call; do the same for any future tool stories.

2. **Atelier asymmetries are first-class concerns.** `docnames` lists files that `/doc/{name}` 404s on (CSP static assets). `RunTest` requires a global the docs imply is optional. The `case` flag's documented default differs from server behavior. **Pattern**: when wrapping an Atelier endpoint, always write a per-class anti-pattern note in the tool description — these surface during real use, not during tests.

3. **The bootstrap auto-upgrade should also re-run idempotent setup steps, not just redeploy classes.** This was the trigger for the `EnsureUnitTestRoot` fix being hard-coded into both bootstrap and per-call. If the user-level CLAUDE.md rule was already in place, we'd have toggled the VSCode extension during the %SYS export too — a similar "run on every connection, not just first-install" defensive pattern.

4. **Per-call defensive setup (run + capture-and-continue) > one-time bootstrap setup for namespace-scoped state.** `^UnitTestRoot` is per-namespace. The bootstrap only runs in the configured namespace. Per-call ensure in the target namespace is what made the fix actually work for the original `USER`-namespace failure case.

5. **Code-review agents catch real defects when given specific focus areas.** `cr-10-2`'s short-path collision finding wasn't on my radar; the review prompt mentioned "concurrency correctness" and "path-traversal safety" and that was enough for the reviewer to find the data-loss surface. Keep prompts directive about what to look for.
