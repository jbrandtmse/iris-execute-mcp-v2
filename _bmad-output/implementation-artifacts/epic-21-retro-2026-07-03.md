## What Went Well

- **"Nice integration of a new feature"** (Project Lead). Two-story epic, zero HIGH findings across both code reviews, clean-room discipline maintained end-to-end.
- **Rule #16 (verify-before-trusting spec claims) paid for itself three separate times at story creation**, before any dev work started: the proposal's "embedded SQL" claim was wrong (the bootstrap deploys to non-interop namespaces — dynamic SQL was required instead, Amendment A1); the proposal's "`gen-bootstrap.mjs` uses a glob" claim was wrong (it's an explicit ordered array — Amendment A2); and the test package name was wrong (`ExecuteMCPv2.Tests`, plural, not `ExecuteMCPv2.Test`, Amendment A3). All three were caught and corrected in `epics.md` before dev began, not discovered mid-implementation.
- **Mid-epic stakeholder decision handled cleanly.** CR 21.0-1 (Queue-fallback false-pairing on empty `ReturnQueueName`) was surfaced by code review as a deferred item, then resolved via a direct question to the stakeholder at Story 21.1 creation — the answer (require both non-empty) became AC 21.1.9 / improvement I3, implemented, tested, and closed in the same epic rather than accumulating in the ledger.
- **The episode-compression algorithm — the epic's stated highest-risk component (per the sprint-change-proposal) — shipped with zero HIGH findings** and a full §6.4 edge-case fixture ledger (abandoned-request unwind, loop-count-in-signature, trace-event attachment, queue-rooted episodes, envelope-wrapper peel, interleaved-no-collapse, errored-iteration-split-and-resume).
- **Live smoke evidence was real, not simulated**: every claimed live-HTTP or live-render check in this epic was backed by an actual Mermaid 11 render (via chrome-devtools-mcp, `RENDER_OK` assertions) or an actual HTTP round-trip against HSCUSTOM/SADEMO — not a description of what a smoke *would* show.

## What Didn't Go Well (Process Pattern, Not a Person Problem)

- **Three real defects slipped past the entire automated pyramid and the story-mandated smoke, surfacing only when the Project Lead independently pushed the tool against real data on a second namespace (SADEMO) and then a full namespace walkthrough.** None were logic bugs — all three were *environment-shape* bugs:
  1. Cross-session dedup was inert on every real trace (anomaly-warning comments embed each session's own message row-id, so structurally identical flows never compared byte-equal).
  2. Error summaries rendered raw, locale-prefixed kernel boilerplate on SADEMO specifically, because that namespace lacks the Ens message-table mapping HSCUSTOM has.
  3. The SAME session's error text varied between REST worker processes, because `GetErrorText`'s message-table language selection happens per-process.
  All three were fixed same-day with regression tests once found — but ~90 green ObjectScript tests, 0 HIGH across two code reviews, and the AC-mandated HSCUSTOM-only smoke did not and structurally could not have caught any of them. → **Rule #34 codified.**
- **`iris_execute_tests` returned an early partial snapshot (a subset of tests, all passing, 0 failures) in every single pipeline stage of both stories** — dev, QA, and code review each independently rediscovered this and lost a round-trip re-verifying per-class before trusting the result. Never a false failure, always a silently-incomplete "success." → **Rule #35 codified.**
- **The deferred-work ledger has grown to 14 open items across three consecutive feature epics** (19 → 20 → 21), each retro-review gate correctly re-deferring them because the current epic wasn't the designated cleanup epic. That's a legitimate process — but the backlog is no longer small.

## Key Insights

1. **Test-shape blindness is namespace-shaped, not logic-shaped.** A tool whose correctness depends on IRIS state that varies by namespace (schema presence, message-table mappings) or worker process (locale selection) needs its live-smoke AC to specify a second, differently-shaped environment — "smoke passed on HSCUSTOM" and "smoke passed on real diverse data" are different bars, and this epic's ACs only asked for the first one.
2. **Rule #16 verification at story creation is cheap insurance.** All three amendments this epic (A1/A2/A3) were caught before a single line of dev code was written, by the simple discipline of reading the actual generator script and class package instead of trusting the proposal's claims.
3. **A tool-quirk that recurs across every pipeline stage is a rule, not a coincidence.** `iris_execute_tests`'s partial-snapshot behavior wasted time three times over two stories before anyone thought to write it down as a checkable rule rather than a one-off surprise.
