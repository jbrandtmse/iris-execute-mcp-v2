## What Went Well

1. **The Epic-14 foundation paid off on first real use.** The `mutates`/default-seed machinery (built but unexercised in Epic 14 — its #1 watch-item) was driven by 5 real tools and worked: every new write action ships default-disabled, every read enabled, proven through the real `McpServerBase.handleToolCall` gate in each story.
2. **Adversarial 3-layer review earned its cost — caught 5 HIGHs the dev AND the Acceptance Auditor missed.** Three were security: the `.nullable()` `action:null` per-action governance fail-open (15.0), the `purge` wildcard-only full-wipe bypass (15.4 — the Auditor rated the AC "satisfied" on the literal `confirm + ≥1 bound` check; the Blind/Edge hunters caught that `"*"` semantically counts as no bound), and the partial SQL grant + double-render on a mixed privilege list (15.5). The split (literal-AC auditor vs. semantic hunters) is exactly why all three surfaced.
3. **Live-endpoint smokes confirmed the security fixes end-to-end.** The lead smoke hit the deployed REST routes over real HTTP and asserted the destructive paths were REJECTED on the running server — wildcard purge refused with "bounded scope" (15.4), `grant SELECT,BOGUS` refused before `SaveObjPriv` (15.5). Mocked-fetch unit tests cannot show a live guard holding. → codified as **Rule #26**.
4. **Empirical IRIS API probing (Rules #14/#16) corrected the plan repeatedly, no fabrication.** `iris_ldap_manage` `test` → no connection API exists → honestly scoped to config-validity (15.2); `iris_audit_manage` enable/disable → `Security.System.Modify(AuditEnabled)` not the Internal `Stop()`, purge → bounded `%SYS.Audit.Delete` not unbounded `Erase` (15.4); `iris_x509_manage` import → object-create+Save not file-based `Import`, and the handler deliberately avoids `GetProperties` because it populates private-key material (15.3); `iris_resource_manage` column-level → `%SQL.Manager.API.SaveObjPriv` not `%SYSTEM.SQL.Security` (15.5).
5. **Secret hygiene held under adversarial test.** LDAP bind-password (15.2) and X.509 private-key material (15.3) never appear in output — enforced server-side, verified by recursive deep-scan tests proven NON-VACUOUS (they throw on hostile leaking payloads) and by live `get` paths.
6. **Back-compat discipline (Rule #19) held while EXTENDING a live tool.** Story 15.5 added SQL-privilege actions to the existing `iris_resource_manage` with a negative-leakage proof (stray fields never reach the resource wire body) and the existing handlers byte-for-byte untouched.

## What Could Be Better (watch-items)

1. **Two planning assumptions were wrong and only the first real consumer falsified them.** Epic 14 retro AI#4 ("regenerate `governance-baseline.ts` whenever a tool is added") was self-contradictory with default-disable; the epics.md "single BOOTSTRAP_VERSION bump at the closing story" was incompatible with the content-hash drift test. Both forced mid-epic lead decisions (Story 15.1 halted twice with `## Clarification Needed`). → the lessons are codified (**Rules #23, #24**), and the cross-epic reach is flagged for Epics 16/17 (same "one bump at closer" language).
2. **The frozen baseline generator is a live footgun.** `gen-governance-baseline.mjs` regrows the frozen 141-key baseline when run for ANY reason; the lead tripped it fetching counts during Story 15.6 prep and had to `git checkout`. Story 15.1's prose "do not re-run" note was insufficient. → **Rule #25** + a deferred `--check`-mode implementation (routed to a CI story).
3. **The Acceptance Auditor's literal-AC reading missed a semantic security hole (15.4).** Not a process failure (the hunters caught it), but a reminder that "AC text satisfied" ≠ "AC intent satisfied" for security ACs — the wildcard-as-bound gap lived between the literal check and the intent.
4. **Per-story bootstrap version churn.** `BOOTSTRAP_VERSION` moved 5× across the epic (one per ObjectScript story). This is correct (it's a content hash) but means intermediate commits carry distinct versions; the closing story confirms the final consolidated value. Documented in Rule #24 so it isn't mistaken for drift.
