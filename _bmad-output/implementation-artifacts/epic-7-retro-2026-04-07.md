## What Didn't Go Well / Challenges

1. **MCP `structuredContent` protocol violation** — All tool files in iris-data-mcp returned raw arrays as `structuredContent`, violating the MCP protocol requirement that it must be a JSON object (record). This affected docdb.ts, analytics.ts, and rest.ts. Unit tests didn't catch it because mock contexts don't enforce MCP protocol schema rules. Found only during end-of-epic MCP tool testing.

2. **Pre-existing bugs surfaced during tool sweep** — `webapp.get` URL-encoding issue (from Epic 4) and `production.messages` missing filter validation (from Epic 5) were both found and fixed during the end-of-epic sweep. These had been in the deferred work file but not resolved until live testing exposed them.

3. **MCP server restarts require manual user intervention** — After fixing the `structuredContent` bug, the MCP servers needed to be restarted to pick up the rebuilt dist. This can't be automated from within Claude Code, creating a friction point in the test-fix-verify cycle.

4. **DocDB integration tests 100% skipped** — `%Service_DocDB` is disabled on the test IRIS instance, so all 9 DocDB lifecycle tests are skipped. The DocDB tool suite has never been tested end-to-end against a live IRIS DocDB API. Unit tests cover the TypeScript logic but not actual IRIS behavior.

## Lessons Learned

1. **End-of-epic MCP tool testing is the irreplaceable safety net** — Third consecutive epic where it caught bugs that unit tests and curl testing missed. The `structuredContent` array bug was invisible to both unit tests (mocks don't enforce protocol) and curl (raw HTTP doesn't go through MCP SDK). Only the actual MCP protocol path catches MCP-level violations.

2. **Built-in IRIS APIs should be preferred over custom handlers** — When IRIS provides a REST API (DocDB, Management, Atelier), use it directly. Custom ObjectScript handlers are only needed when no built-in API exists (DeepSee analytics, security, interop). 2 out of 3 feature stories avoided handlers entirely.

3. **`toStructured()` should be applied proactively** — Any tool that could return a non-object (arrays, primitives) from an external API must wrap the result. This should be a standard pattern in tool handler boilerplate, not discovered after the fact.

4. **The epic-cycle pipeline is production-grade for feature epics** — Seven epics, 48 stories, fully autonomous story creation through commit. The pattern scales and the quality metrics hold. Zero user intervention needed during story development phases.

5. **Deferred work triage should be aggressive and decisive** — Formally closing won't-fix items (with rationale) is better than keeping a growing list of "maybe someday" items. The file is useful only when it's short and actionable.
