## What Went Well

1. **Handler pattern scaled perfectly** — The NamespaceManage/NamespaceList pattern from Story 4.2 was reused cleanly across all 8 feature stories. Config.cls (6 methods) and Security.cls (15+ methods) required zero architectural rework.

2. **Shared infrastructure investment paid off** — Moving booleanParam and test helpers to @iris-mcp/shared in Story 4.0 meant iris-admin-mcp had zero duplication from day one. Every tool story imported from shared without friction.

3. **Consistent tool naming and annotations** — All 22 tools follow `iris.<domain>.<action>` convention with correct scope (SYS/BOTH), destructiveHint, and readOnlyHint annotations. Discoverable and predictable.

4. **Story sequencing prevented rework** — Building foundations first (4.0 cleanup, 4.1 package setup, 4.2 Config handler) before security tools (4.4-4.8) meant no backtracking.

5. **Password/secret security discipline** — NFR6 compliance verified across all tools. 541 tests include explicit checks that passwords and OAuth2 client secrets never appear in responses.

6. **Integration test pattern reuse** — Story 4.9 mirrored iris-dev-mcp integration test pattern exactly, with dependency-ordered creation and reverse-ordered cleanup. 28 tests, zero pattern confusion.

7. **Strong velocity** — 10 stories (22 tools) completed in a single epic cycle with consistent quality.

## What Didn't Go Well / Challenges

1. **3 bugs found only through live testing — again** — Same lesson as Epic 3 (9 bugs then). Unit tests with mocked HTTP cannot catch:
   - Wrong IRIS API usage (`Config.Namespaces.NamespaceList()` doesn't exist as a class method — it's a `%ResultSet` query)
   - Namespace switching crash (`New $NAMESPACE` makes `ExecuteMCPv2.Utils` invisible in %SYS catch blocks)
   - SQL column name mismatch (`Resource_` vs actual projection name)

2. **`New $NAMESPACE` pattern was fundamentally unsafe** — Used in every method of both Config.cls (6 methods) and Security.cls (15+ methods). If any system class call threw an exception, the catch block crashed because `ExecuteMCPv2.Utils` only exists in HSCUSTOM, not %SYS. The working Security methods only worked by luck (happy path never threw).

3. **IRIS unavailable during Story 4.8** — OAuth2 handler couldn't be compiled or tested on IRIS. TypeScript tests passed but live verification was impossible. Deferred to hotfix.

4. **`iris.doc.load` document name mapping issues** — The bulk load tool mapped file paths to wrong document names (e.g., `REST.Config.cls` instead of `ExecuteMCPv2.REST.Config.cls`). Required delete + re-upload + force compile workaround during hotfix.

5. **ObjectScript classes created directly on IRIS** — During debugging, a temporary `ConfigDebug.cls` was created via `iris.doc.put` without a disk file, bypassing source control. Had to be manually cleaned up.

6. **48 deferred items accumulated** — Across 4 epics, deferred work grew to 48 items. Many are resolved or cosmetic, but the list was never scrubbed. Debt visibility degraded.

## Lessons Learned

1. **Live testing is still non-negotiable** — Third epic confirming this. Added Step 2.5 (Lead Live Verification) to the epic-cycle pipeline as a mandatory gate between dev and code review.

2. **`New $NAMESPACE` must never be used in REST handlers** — Use `Set tOrigNS = $NAMESPACE` / `Set $NAMESPACE = "%SYS"` / `Set $NAMESPACE = tOrigNS` pattern instead. Codified in CLAUDE.md ObjectScript rules.

3. **Config.* classes use `%ResultSet` queries, not class methods** — `Config.Namespaces.NamespaceList()`, `Config.Databases.DatabaseList()`, and `Config.Map*.List()` don't exist as class methods. Must use `##class(%ResultSet).%New("Config.Namespaces:List")` pattern.

4. **All ObjectScript must be on disk first** — Never create production classes directly on IRIS via `iris.doc.put`. Always edit `.cls` files on disk, then deploy with `iris.doc.load`. Updated tool descriptions to make this explicit.

5. **Deferred work needs periodic scrubbing** — 48 items across 4 epics is too many. Story 5.0 will do an aggressive cleanup: fix real bugs, extract duplicated code, add missing validation, and scrub resolved items from the list.

6. **Empty string falsy guards cause bugs** — `if (value)` in ObjectScript and TypeScript skips empty strings on modify operations. Always use `!== undefined` (TypeScript) or `'= ""` with explicit checks (ObjectScript) for optional string fields.
