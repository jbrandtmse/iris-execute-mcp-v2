## What Went Well

1. **Full autonomous pipeline execution** — All 9 stories completed through create -> dev -> live verify -> code review -> commit with zero user intervention until end-of-epic MCP server retest. Pipeline maturity confirmed across two consecutive epics.

2. **Step 2.5 caught 5 bugs inline** — `$ZU(190,28/29)` doesn't exist, `SYS.Database` SQL table doesn't exist, `freeSpaceMB` was actually bytes, lock Owner field format mismatch, `Config.config.Open()` returns CPF path not %Status. All fixed before code review.

3. **Lead-provided verified APIs eliminated fabrication** — The lead queried `%Dictionary.MethodDefinition` and ran live tests before every story, providing only verified API signatures in story context. Result: zero fabricated API names (down from 5/8 stories in Epic 5). Single biggest process improvement of the project.

4. **Zero hotfixes** — Epic 5 had 6 hotfixes. Epic 6 had 0. The combination of verified APIs + Step 2.5 live testing + adversarial code review eliminated post-story fixes entirely. First epic to achieve this.

5. **Handler pattern scaled for third consecutive epic** — Monitor.cls grew to 11 methods, plus two new handler classes (Task.cls, SystemConfig.cls), all following the established save/restore namespace pattern with zero architectural rework.

6. **IPM module.xml updated in-story** — Per Epic 5 retro lesson, Stories 6.2, 6.6, and 6.7 all added handler classes to module.xml in the same commit. Third consecutive epic with compliance.

7. **Four fully functional MCP servers** — 78 tools across iris-dev-mcp, iris-admin-mcp, iris-interop-mcp, iris-ops-mcp. All 34 tested live via MCP protocol at end of epic with zero errors.

## What Didn't Go Well / Challenges

1. **IRIS API discovery remains the main risk** — Even with verified APIs, 4 out of 7 feature stories hit at least one wrong API assumption during development. The dev agent still guesses when story context doesn't cover an edge case. Step 2.5 is the safety net, not the prevention.

2. **MCP tool testing couldn't happen inline** — The iris-ops-mcp server wasn't connected to Claude Code until the user restarted MCP servers at the end. Step 2.5's "MCP tool testing" layer only worked via curl, not actual MCP protocol. The 34-tool retest at the end was the real MCP verification.

3. **Deferred work grew to ~40+ items** — 12 new items added across 6 code reviews. Most are LOW priority. Items from Epics 1-3 will likely never be addressed. The file is becoming noise rather than actionable.

4. **bootstrap-classes.ts not updated with Epic 6 classes** — Story 6.0 regenerated with 9 classes from Epics 1-5, but the 3 new Epic 6 handler classes (Monitor.cls, Task.cls, SystemConfig.cls) aren't included. Must be done in Story 7.0.

## Lessons Learned

1. **Lead-provided verified APIs is the breakthrough pattern** — Epic 5: 5/8 fabricated. Epic 6: 0/7 fabricated. The pattern: lead queries `%Dictionary.MethodDefinition`, runs live tests, includes only verified signatures in story context. Continue as-is — formalization not needed.

2. **Step 2.5 curl testing and end-of-epic MCP testing are complementary** — Curl catches IRIS API errors (wrong methods, missing tables, return type mismatches). MCP protocol testing catches TypeScript client-side errors (response parsing, dual-JSON, content-type). Both layers required.

3. **Zero hotfixes is achievable and should be the baseline** — Verified APIs + Step 2.5 + adversarial code review = zero hotfixes. This should be the expectation for all remaining epics.

4. **Config.* classes have non-standard return signatures** — `Config.config.Open()` returns CPF path as output param, not %Status. `Config.NLS.Locales.Get("current")` doesn't exist. IRIS Config classes are inconsistent — always test live before coding.

5. **Deferred work needs periodic aggressive triage** — Items accumulate faster than they're resolved. Formally closing won't-fix items reduces noise and keeps the file useful as an actionable backlog.
