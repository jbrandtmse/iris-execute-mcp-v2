## 2. What Went Well

1. **The Rule #52 scope-seam repeated cleanly across 30.2 → 30.3.** 30.2 shipped a MINIMAL README stub (measurement table + a pointer) with the deferred docs-enrichment seam named explicitly; 30.3 enriched it (roster tables, visibility-vs-governance layering rules, client-config env rows) without re-measuring or re-creating the section, and removed the stub marker. Zero rework — the seam that worked for Epic 29's 29.0→29.1 held again for a docs deliverable.
2. **The Opus 3-layer adversarial review caught a real, non-obvious MED that dev AND QA both missed.** 30.2 CR-30.2-1: the `toolVisibility` counts were computed at construction but went STALE after the dynamic `addTools`/`removeTools` paths — the Edge Case Hunter found it; the fix wired a `recomputeToolVisibilityCounts()` helper into both paths with +2 regression tests. A green suite + a passing acceptance audit did not surface it; the adversarial edge-case lens did.
3. **Verify-before-applying rejected a source-blind false positive.** In 30.3 the Blind Hunter (working from the diff only) flagged a "MED" that the admin `core` prompt-note omitted `iris_role_list` from the core-hidden set. The review coordinator checked `presets.ts` directly, found `iris_role_list` is in admin `core.include` (core-VISIBLE — spec §2.5 confirms), and correctly DISMISSED the finding — applying the suggested patch would have INJECTED a factual error. This is Rule #16/#47's discipline (verify the claim against source) operating inside the review layer itself.
4. **Additive discipline proven mechanically, not asserted.** Default `full`/unset ⇒ byte-for-byte today's `tools/list` was proven on the BUILT dist for all 5 servers (capstone smoke + the Rule #19 back-compat test); frozen baseline `1e62c5ad5bf7` + `BOOTSTRAP_VERSION` untouched across all four stories; `gen:governance-baseline:check` exit 0 every story; Rule #31 held (no tool/prompt counts moved anywhere).
5. **QA added exactly the one test that mattered for a docs story — a doc-rot guard.** The README/per-server-READMEs now hard-code roster + payload counts; existing roster tests derive ground-truth from the same `toolPresets` object the docs were transcribed from, so they were blind to a docs/rosters split. QA's new `docs-visibility-roster-sync.test.ts` re-derives counts LIVE and diffs them against the documented numbers (mutation-verified); the review then EXTENDED it (10→17 tests) to cover the per-server READMEs + `iris-mcp-all` table + CHANGELOG after the Edge Case Hunter flagged the guard's original root-README-only scope (MED-1).

## 3. What Could've Gone Better

1. **The code-review sub-agent repeatedly stalled in a poll-loop instead of triaging its own completed layers.** On BOTH 30.2 and 30.3, the review coordinator spawned its three layers (Blind Hunter / Edge Case Hunter / Acceptance Auditor), then returned "waiting on a background poller" rather than reading the layers' completed output files and triaging — requiring a lead `SendMessage` nudge (pointing it at the on-disk layer outputs) to finalize each time. No quality impact (the nudge + the lead's own consolidation caught everything, and every finding was independently re-verified against source before applying), but it added a manual round-trip per story. **Disposition:** harness/process friction, NOT a codebase class-of-bug — recorded here, not codified (Project Lead decision, §5). Mitigation for future runs: the lead can pre-instruct the review spawn to read layer output files directly on completion rather than block on a poller.
2. **Minor doc-accuracy drift caught late (both in 30.3 review, both LOW).** The ops README prompt-note over-included `iris_ecp_status` (a tool the `check-system-health` prompt references only as an *area*, not the tool), and the admin developer-roster PROSE enumerated 9 tools against a count of 10. Both were patched; both are the kind of hand-transcription slip the new doc-rot guard now mechanically prevents for the COUNTS, though not for prose tool-name lists. Accepted residual: prose tool-name references in prompt-limitation notes remain hand-verified.
